# Contributing to Stellar UI

Welcome! We appreciate your interest in contributing to Stellar UI. To keep the codebase clean, performant, and maintainable, please adhere to the following architectural guidelines.

## Workflow (fork model)

Stellar is developed across sibling repos — `stellar-api`, `stellar-ui`, `stellar-compose` — using a **fork model** with a single `main` trunk.

**Remotes:** `origin` = your fork (`<you>/stellar-ui`), `upstream` = `orphic-inc/stellar-ui` (canonical). Never push to `upstream`; PRs go fork → upstream.

```bash
git clone git@github.com:<you>/stellar-ui.git && cd stellar-ui
git remote add upstream git@github.com:orphic-inc/stellar-ui.git
```

- **`main` is the only long-lived branch and the sole PR target** — there is no `develop` or `staging`. Cut feature branches from a fresh `main` on your fork and PR into `upstream/main`.
- **Rebase-only / linear history.** Upstream accepts **rebase-and-merge only** — a merge commit (e.g. from `git merge main` into your branch) makes GitHub refuse the merge. Keep your branch rebased on `upstream/main`: `git fetch upstream && git rebase upstream/main`.
- **Triage vocabulary.** Issues use `needs-triage` / `needs-info` / `ready-for-agent` / `ready-for-human` / `wontfix`; target `--repo orphic-inc/stellar-ui` explicitly (your `origin` is a fork). The shared API/UI contract-hygiene lifecycle these steps reference is **stellar-api ADR-0018** (a cross-system decision; it lives in the stellar-api repo, not here).

## Code Standards

### Styling

We have standardized on **TailwindCSS** for styling.

- Do not add new SCSS or CSS files unless absolutely necessary for complex animations or global reset behavior that Tailwind cannot handle.
- Utilize Tailwind utility classes directly on React components.

### State Management

We use **Redux Toolkit (RTK)**.

- Data fetching and caching should utilize RTK Query where applicable.
- Global application state should be partitioned into clean slices (`createSlice`).
- Avoid deeply nested or overly complex component-level state if it needs to be accessed by multiple components.

### Type Safety

The UI relies heavily on the OpenAPI schema generated by the `stellar-api` backend.

- Do not manually write API response interfaces. Derive types from the generated spec via `components['schemas']['...']` or `paths[...][method][...]`.
- When stellar-api changes a response shape or adds a route, regenerate types with **`api:sync`** (not `api:generate` — that only reruns codegen over the already-vendored spec and won't pull the change):
  ```bash
  # stellar-api must be a sibling directory (../stellar-api)
  npm run api:sync
  ```
  This copies the fresh `openapi.json` from `../stellar-api` into `src/types/openapi.json`, runs `openapi-typescript` against it, and reformats. Commit both `src/types/api.ts` and `src/types/openapi.json`.
- Run `npx tsc --noEmit` after regenerating to catch type regressions before committing.
- Use these generated types throughout your RTK Query services, Redux thunks, and component props.

### Component Structure

- Keep components small and focused on a single responsibility.
- Place reusable visual components in a common UI or components directory.
- Container components (which connect to Redux or API layers) should be kept separate from purely presentational components.

## Submitting Pull Requests

1. `npm run format` — format all of `src/`
2. `npm run lint` — must be clean on new/changed files
3. `npm run typecheck` — must pass
4. `npm run test` — full suite must pass
5. For changes touching critical user paths, also run the Playwright end-to-end suite in `e2e/`: `npm run test:e2e` (against a running app at `BASE_URL`, default `http://localhost:9000`).
6. If the PR responds to a stellar-api change that touched response shapes: run `npm run api:sync`, commit `src/types/api.ts` (and `src/types/openapi.json`), and link the API PR.
7. When stellar-api ships a new UI-consumable surface (a new route, a response-shape change, or a new OpenAPI schema), confirm a paired stellar-ui tracking issue exists for it — file one if it doesn't. A backend surface with no UI issue is how features ship half-built and untracked. (Note: a route that isn't registered in stellar-api `src/lib/openapi.ts` is invisible to `api:generate` and cannot be consumed type-safely — that gap belongs on the API side first.)
8. Provide a clear description of the visual or logical changes in your PR.

> **Adding a dependency (or subpath import) that resolves through a conditional `exports` map.** Packages whose `package.json` resolves through a conditional `exports` map — and subpath imports into them, e.g. `import 'katex/dist/katex.min.css'` — can't be followed by the `import/no-unresolved` resolver **Codacy Static Code Analysis** runs, so the Codacy PR check fails with an unresolved-import issue even though the import type-checks and runs fine. Local `npm run lint` can pass — the local node resolver reads the package's `main` field — so a green local lint is **not** proof the Codacy check will pass. The fix is the established one: add a pattern for the module to the `ignore` array of the `import/no-unresolved` rule in `.eslintrc.js` (that list exists for exactly this class; use `^pkg/` to also cover subpath imports). Verify with `npm run lint`, and confirm the Codacy check on the PR goes green.
