# Contributing to Stellar UI

Welcome! We appreciate your interest in contributing to Stellar UI. To keep the codebase clean, performant, and maintainable, please adhere to the following architectural guidelines.

## Code Standards

### Styling

We have standardized on **TailwindCSS** for styling.

- Do not add new SCSS or CSS files unless absolutely necessary for complex animations or global reset behavior that Tailwind cannot handle.
- Utilize Tailwind utility classes directly on React components.

### State Management

We use **Redux Toolkit (RTK)**.

- Data fetching and caching should utilize RTK Query where applicable.
- Global application state should be partitioned into clean slices (`createSlice`).
- Avoid deeply nested or overly complex component-level state if it needs to be accessed by multiple components.

### Type Safety

The UI relies heavily on the OpenAPI schema generated by the `stellar-api` backend.

- Do not manually write API response interfaces. Derive types from the generated spec via `components['schemas']['...']` or `paths[...][method][...]`.
- When stellar-api changes a response shape or adds a route, regenerate types:
  ```bash
  # stellar-api must be a sibling directory (../stellar-api)
  npm run api:generate
  ```
  This exports `openapi.json` from stellar-api, runs `openapi-typescript` against
  it, and reformats the output. Commit the resulting `src/types/api.ts`.
- Run `npx tsc --noEmit` after regenerating to catch type regressions before committing.
- Use these generated types throughout your RTK Query services, Redux thunks, and component props.

### Component Structure

- Keep components small and focused on a single responsibility.
- Place reusable visual components in a common UI or components directory.
- Container components (which connect to Redux or API layers) should be kept separate from purely presentational components.

## Submitting Pull Requests

1. `npm run format` — format all of `src/`
2. `npm run lint` — must be clean on new/changed files
3. `npm run typecheck` — must pass
4. `npm run test` — full suite must pass
5. For changes touching critical user paths, also run the Playwright end-to-end suite in `e2e/`: `npm run test:e2e` (against a running app at `BASE_URL`, default `http://localhost:9000`).
6. If the PR responds to a stellar-api change that touched response shapes: run `npm run api:generate`, commit `src/types/api.ts`, and link the API PR.
7. When stellar-api ships a new UI-consumable surface (a new route, a response-shape change, or a new OpenAPI schema), confirm a paired stellar-ui tracking issue exists for it — file one if it doesn't. A backend surface with no UI issue is how features ship half-built and untracked. (Note: a route that isn't registered in stellar-api `src/lib/openapi.ts` is invisible to `api:generate` and cannot be consumed type-safely — that gap belongs on the API side first.)
8. Provide a clear description of the visual or logical changes in your PR.
